一、系统用例图（Use Case Diagram）

建模说明：系统按真实仓库拆分为 Gateway + Auth + Problem + Judge + Blog + Chat 六类核心能力。普通用户在题目、判题、博客、AI 辅助四条主线上交互；题目管理员和系统管理员承担治理职责；Go-Judge 与 DashScope 分别作为外部判题执行器与大模型服务参与扩展流程。

二、核心用例规约：提交代码判题（UC-JUDGE-001）

用例名称提交代码判题
主要参与者普通用户（主）、网关、判题服务、题目服务、Go-Judge（辅）
触发条件用户在题目详情页点击“提交代码”
前置条件
  1. 用户已登录且 Token 有效（Redis 白名单存在）
  2. 题目存在且可访问
  3. 语言已启用
后置条件
  1. 生成 submission 记录
  2. 异步完成判题并更新状态、得分、通过率
  3. 用户可在提交记录中查询结果
基本事件流
  1. 用户提交 problemId + languageId + code
  2. 网关解析 JWT，注入 X-User-Id 后转发至 /judge/submit
  3. 判题服务调用 Problem 服务校验题目与语言
  4. 创建 submission(status=0)，返回“已提交”
  5. 异步任务将状态置为 Judging，加载测试用例与限制
  6. 按语言选择编译/解释执行策略，循环运行测试用例
  7. 汇总时间、内存、通过率，写回最终状态（AC/WA/TLE/MLE/RE/CE/PA/SE）
备选流
  1. A1 编译失败：状态更新为 CE，写入 compileMessage
  2. A2 某用例运行异常：状态更新为 RE/TLE/MLE/OLE
  3. A3 题目或语言不存在：同步返回 400，不创建记录
  4. A4 测试用例缺失：状态更新为 SE，提示配置异常
异常流
  1. Token 无效/过期：网关直接返回 401
  2. Go-Judge 无响应：判题服务捕获异常并标记 SE
  3. 数据库写入失败：事务回滚，返回失败消息
业务规则
  1. 提交记录采用逻辑删除字段，默认只查未删除数据
  2. Java/Python 在运行阶段给予额外资源倍率
  3. WA 且存在部分通过时，状态可升级为 PA（部分通过）
非功能要求
  1. 提交接口应快速返回（异步判题）
  2. 每次判题必须清理 Go-Judge 缓存文件
  3. 服务间调用经 Nacos + Feign + LoadBalancer 发现与转发
