## 1. 引言

### 1.1 编写目的

为了高效、有序地完成 EmiyaOJ-Cloud 在线判题系统的实训开发与交付，特制定本系统实施计划。本计划用于明确项目实施范围、人员分工、开发管控方式、阶段进度、技术框架、测试验收要求和主要风险，为小组成员在需求分析、设计、编码、测试、部署和演示阶段提供统一依据。

### 1.2 项目背景

EmiyaOJ-Cloud 是一个基于前后端分离与 Spring Cloud 微服务架构的在线判题系统，面向在线编程练习、题目管理、代码提交、自动判题、博客交流和 AI 辅助问答等场景。完整项目包含两个前端应用：管理端和用户端。当前仓库主要体现后端微服务工程结构，包含认证、题目、判题、博客、聊天、审核、网关、公共模块以及 Docker Compose 部署编排，两个前端项目未放置在当前仓库中，但仍属于系统实施、联调、测试和演示范围。项目实施过程中还使用 Jenkins 进行流水线部署，用于自动拉取代码、执行 Maven 构建、构建 Docker 镜像并发布到测试/演示环境。

本项目作为大学生软件工程实训小组作业，实施目标不是从零搭建简单示例系统，而是在现有代码基础上完成系统梳理、功能完善、联调测试、部署演示与文档交付。

### 3.4 Jenkins 流水线管控

项目使用 Jenkins 进行持续集成与流水线部署，主要用于减少手动部署错误，保证演示环境与代码仓库版本一致。

| 阶段 | 主要内容 | 说明 |
| --- | --- | --- |
| 代码拉取 | 从 Git 仓库拉取指定分支代码 | 默认使用 `main` 或 `develop` 分支 |
| 编译构建 | 执行 Maven 构建命令 | 编译父工程和所有微服务模块 |
| 单元测试 | 执行关键模块测试 | 覆盖认证、题目、判题核心逻辑 |
| 镜像构建 | 根据各服务 Dockerfile 构建镜像 | Gateway、Auth、Problem、Judge、Blog、Chat、Moderation 等服务 |
| 镜像发布 | 推送至镜像仓库或保存在部署服务器本地 | 实训环境可使用本地镜像方式 |
| 容器部署 | 使用 Docker Compose 启动或更新服务 | 包含基础设施和业务微服务 |
| 部署验证 | 检查容器状态、端口、Nacos 注册和核心接口 | 失败时查看 Jenkins 控制台日志和容器日志 |

流水线建议包含以下基本步骤：

```text
checkout -> mvn clean package -> docker build -> docker compose up -d -> health check
```

## 5. 项目技术框架

### 5.1 总体架构

项目采用前后端分离与 Spring Cloud 微服务架构。前端分为管理端和用户端：管理端面向管理员、教师或运维人员，用户端面向普通刷题用户。两个前端通过 HTTP API 统一访问 Gateway，网关完成路由和认证上下文注入，再转发到 Auth、Problem、Judge、Blog、Chat、Moderation 等业务服务。各服务通过 Nacos 注册发现，通过 OpenFeign 进行服务间调用，通过 MySQL 独立数据库保存业务数据，通过 Redis 保存 Token 白名单，通过 RabbitMQ 处理博客审核异步任务，通过 MinIO 保存博客图片，通过 Go-Judge 进行代码编译和运行。

## 7. 部署实施计划

### 7.1 本地开发环境

| 工具 | 建议版本 |
| --- | --- |
| JDK | 21 |
| Maven | 3.9.x |

### 7.2 部署步骤

1. 检查 JDK 21、Maven、Docker 是否安装完成。
2. 在项目根目录执行 Maven 构建，确保各模块可以编译。
3. 使用 SQL 脚本初始化数据库，或使用 `Dockerfile.mysql` 自动构建初始化镜像。
4. 启动 Docker Compose 中的基础设施：MySQL、Redis、Nacos、RabbitMQ、MinIO、Go-Judge。
5. 启动 Gateway、Auth、Problem、Judge、Blog、Chat、Moderation 服务。
6. 在 Nacos 控制台确认服务已注册。
7. 使用 Swagger 或 Postman 完成登录、题目、判题、博客、审核等接口联调。
8. 准备演示账号、题目、测试用例、提交代码和博客内容。

### 7.3 Jenkins 流水线部署步骤

1. 在 Jenkins 中配置 JDK 21、Maven、Git、Docker 或 Docker Compose 环境。
2. 创建 Pipeline 或 Freestyle 流水线任务，绑定项目 Git 仓库地址和构建分支。
3. 配置凭据，包括 Git 仓库凭据、镜像仓库凭据、服务器 SSH 凭据以及必要的环境变量。
4. 执行 Maven 构建，生成各微服务可运行 Jar 包。
5. 拉取管理端和用户端前端项目，执行前端依赖安装与打包。
6. 根据各服务 Dockerfile 构建业务服务镜像，必要时构建前端 Nginx 镜像或发布静态资源。
7. 使用 Docker Compose 更新容器，保证 MySQL、Redis、RabbitMQ、MinIO、Nacos、Go-Judge、业务服务和前端服务处于运行状态。
8. 部署完成后检查容器状态、Nacos 服务注册情况、Gateway 端口、前端访问地址和核心接口。
9. 保存 Jenkins 构建日志和部署截图，作为实训答辩佐证材料。

| `ADMIN_FRONTEND_REPO`、`USER_FRONTEND_REPO` | 管理端和用户端前端仓库地址 |

## 8. 风险与应对措施

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| 微服务数量多，联调复杂 | 进度延迟 | 先保证 Auth、Gateway、Problem、Judge 主流程，再扩展博客和 AI |
| Docker 环境占用资源高 | 服务启动失败 | 可分批启动基础设施和核心服务，演示时保留必要容器 |
| Go-Judge 沙箱依赖系统权限 | 判题无法运行 | 提前在 Linux 或 Docker Desktop 环境验证，准备备用判题结果截图 |
| Jenkins 环境缺少 Docker 权限 | 流水线无法构建或部署容器 | 提前配置 Jenkins 用户权限，必要时使用部署脚本或 SSH 到部署机执行 |
| Jenkins 凭据或环境变量配置错误 | 构建成功但服务启动失败 | 将敏感配置放入 Jenkins Credentials，并建立环境变量核对清单 |
| 前端项目不在当前仓库 | 实施计划与实际交付范围容易遗漏 | 在交付物中单独列出管理端和用户端前端仓库、构建产物和部署地址 |
| 前后端接口字段不一致 | 页面调用失败或数据展示异常 | 以 Swagger/API 文档为准进行联调，关键接口固定请求和响应格式 |
| AI 或阿里云审核 API Key 不可用 | Chat/Moderation 演示受影响 | 使用环境变量配置，准备异常返回演示或 Mock 数据 |
| JWT 与权限配置错误 | 接口无法访问 | 建立白名单和管理员账号检查清单 |
| 分工边界不清 | 重复开发或遗漏 | 每日同步任务状态，按模块负责人合并代码 |
| 数据库初始化不一致 | 接口报错 | 统一使用 `sql` 目录脚本和演示数据库备份 |

本节根据新下发的《项目总结》目录，对实施计划中与“项目概述、开发计划、可行性分析”相关的内容进行集中补充。原有实施计划仍作为项目执行依据，本节用于后续合并项目总结报告时快速引用。

### 11.1 项目概述补充

#### 11.1.1 项目背景

EmiyaOJ-Cloud 面向高校程序设计课程、算法训练、实验教学和实训答辩场景。传统线下代码作业存在批改效率低、结果反馈慢、过程数据不完整等问题；普通在线练习平台又难以覆盖教师后台管理、竞赛组织、内容审核、AI 辅助问答和自动化部署演示等实训要求。因此，本项目基于现有后端微服务工程，整合在线判题、题目管理、竞赛、博客社区、AI 问答和持续部署能力，形成可演示、可验收、可扩展的在线判题系统。

#### 11.1.2 项目定位

| 项目定位 | 说明 |
| --- | --- |
| 教学训练平台 | 支持教师维护题目、测试用例、语言和题单，学生在线刷题并查看判题结果 |
| 竞赛组织平台 | 支持竞赛创建、报名、题目关联、提交校验和排行榜展示 |
| 社区交流平台 | 支持博客、题解、评论、点赞、收藏和图片上传 |
| 实训演示项目 | 展示 Spring Cloud 微服务、Docker Compose、Jenkins、Go-Judge、RabbitMQ、MinIO 等技术集成能力 |

#### 11.1.3 应用场景

| 场景 | 典型流程 |

| 日常刷题 | 用户登录 -> 浏览题目 -> 提交代码 -> 查看判题结果 |

#### 11.2.6 进度安排

进度安排以第 4 章项目进度计划为准。项目总结中可按“需求分析 -> 概要设计 -> 详细设计 -> 编码实现 -> 接口联调 -> 部署测试 -> 文档答辩”七个阶段展示。

#### 11.2.7 开发预算

本项目为大学生软件工程实训项目，不涉及商业采购预算。预算可按资源投入估算：

| 资源项 | 预算说明 |
| --- | --- |
| 人力投入 | 5 人小组按模块分工完成，主要成本为实训时间 |
| 开发环境 | 使用个人电脑、开源 JDK、Maven、Docker、MySQL、Redis、Nacos、RabbitMQ、MinIO |
| 第三方服务 | AI 和阿里云审核接口按演示账号或免费/低额度测试资源使用 |
| 部署环境 | 本地 Docker Compose 或实验室/个人服务器，Jenkins 可使用本地部署 |

### 11.3 可行性分析补充

#### 11.3.1 技术可行性分析

项目技术栈均为成熟开源或常用工程技术。Spring Boot、Spring Cloud、MyBatis-Plus、MySQL、Redis、Nacos、RabbitMQ、MinIO、Docker Compose 和 Jenkins 均有稳定社区生态；Go-Judge 作为独立沙箱隔离用户代码，满足在线判题安全要求。现有仓库已有模块划分、SQL 脚本、API 文档和部分测试类，具备继续完善、联调和演示的技术基础。

#### 11.3.2 资源可行性分析

项目小组 5 人分工明确，分别负责架构与部署、认证权限、题目竞赛、判题服务、博客/AI/测试。系统可在本地 Docker 环境或轻量演示服务器运行。管理端和用户端虽不在当前后端仓库中，但已纳入实施、联调和验收范围，资源上可通过独立仓库或构建产物交付。

#### 11.3.3 市场可行性分析

在线判题系统在高校程序设计课程、算法训练、竞赛组织和实训教学中具有明确需求。EmiyaOJ-Cloud 相比普通练习系统增加了管理端、博客审核、AI 问答和自动化部署能力，更适合作为课程实训、教学演示和小型团队工程实践项目。若后续继续完善稳定性、题库规模和用户体验，也可扩展为校内教学训练平台。
